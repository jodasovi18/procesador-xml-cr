# Fase 1B-6: Borrador D-150 (IVA mensual) — Diseño

> Documento de diseño (spec). El plan de implementación con pasos TDD va aparte:
> `docs/plans/2026-06-22-fase1b-6-d150.md`.

## Objetivo

Construir el borrador de la **declaración D-150** (IVA mensual) de un cliente para un
período, sobre el resumen tributario clasificación-aware (1B-5):

- **Débito fiscal** = IVA de ventas (por tasa).
- **Crédito fiscal** = IVA de compras deducibles (por tasa), excluyendo
  `TiqueteElectronico` (Decreto 44739-H), No Deducibles y No Sujeto.
- **Liquidación** = débito − crédito, con estado `a_pagar | saldo_favor | cero`.

Incluye **entradas manuales** (ventas y compras sin XML; p.ej. ventas de clientes
sin factura electrónica, o **compras en subastas**). Doble presentación de montos:
**Decimal preciso** (auditoría/reconciliación) y **entero redondeado estilo
OVI-Tribu** (los números que van en el formulario).

**Meta de éxito (esta fase):** `build_d150` + entradas manuales + endpoints, con
golden tests deterministas. La reconciliación completa contra declaraciones reales
(oracle Agrofinca/Solis Feb 2026) se difiere: requiere los XML + `manuales.json`
reales, que **no están en esta máquina**.

## Contexto (sistema viejo)

`build_d150(compras, ventas)` en `parse_xml.py` + `_merge_manuales_d150` en `app.py`:

- **Ventas → débito:** `build_resumen_ventas`, colapsa `{tipo} {pct}` → `{pct}` (suma
  IVA), separa exentas y no_sujetas.
- **Compras → crédito:** excluye `TiqueteElectronico` (`compras_deducibles`),
  `build_resumen` sobre el resto; No Deducibles y No Sujeto **fuera del crédito**;
  informativo de tiquetes (n + IVA excluido).
- **Redondeo OVI:** por tasa `iva = round(base × tasa)` (una sola operación, no suma
  línea por línea), todo a **entero**. Totales = suma de los redondeados.
- **Sin prorrata** (crédito pleno; el oracle de Solis lo confirma como diferencia
  conocida vs. la declaración real).
- **Entradas manuales** (`manuales.json` keyed por período, `{compras, ventas}`):
  - Venta: `tarifa>0` → débito por tasa (`iva = base×tarifa`); `tarifa==0` →
    exentas, o no_sujetas si la marca lo indica.
  - Compra: `tarifa>0 & deducible` → crédito por tasa; `tarifa>0 & !deducible` →
    no_deducibles; `tarifa==0` → exentas o no_sujetas.

## Decisiones de diseño

1. **`build_d150` reutiliza `build_resumen`** (1B-5) para ambos roles — no reimplementa
   la agregación ni la clasificación. Solo colapsa por tasa y arma la liquidación.
2. **Doble presentación.** El motor calcula en **Decimal preciso**; una función aparte
   produce la vista **OVI** (enteros). El endpoint devuelve ambas (`preciso` y `ovi`).
   La vista OVI recomputa `iva_ovi = round(base × tasa)` desde la base precisa (no
   redondea el IVA preciso) y suma los redondeados, fiel al portal.
3. **Tiquetes fuera del crédito** vía un parámetro nuevo `excluir_tipos` en
   `build_resumen` (filtra `Comprobante.tipo_doc`), retrocompatible. El informativo de
   tiquetes (n + IVA) se calcula con una consulta dedicada.
4. **Entradas manuales en una sola tabla** `entradas_manuales` con campo `rol`
   (compra/venta), reflejando el `manuales.json` viejo (`{compras, ventas}`). Se
   mezclan en los mismos acumuladores que los datos ingestados.
5. **Sin prorrata** (crédito pleno), fiel al sistema viejo.
6. La marca **No Sujeto** de una entrada manual es un booleano directo (`no_sujeto`);
   no se reusa `sub_clasificacion`/Combustibles en entradas manuales (más simple, y el
   usuario marca el tratamiento directamente).

## Componentes

### B. Modelo `EntradaManual` (`models/entrada_manual.py`) + migración

| campo         | tipo               | nota                                        |
|---------------|--------------------|---------------------------------------------|
| `id`          | int PK             |                                             |
| `cliente_id`  | FK clientes CASCADE|                                             |
| `periodo`     | str(6)             | YYYYMM                                       |
| `rol`         | str(10)            | `compra` \| `venta`                         |
| `descripcion` | str(255), nullable | nota libre (p.ej. "Subasta ganado 12/05")   |
| `monto`       | Numeric(18,5)      | base (gravado) o monto (exento/no sujeto)   |
| `tarifa`      | Numeric(7,4)       | % IVA; 0 = exento/no sujeto (default 0)     |
| `no_sujeto`   | Boolean            | si `tarifa==0`: True→No Sujeto, False→Exento|
| `deducible`   | Boolean            | solo compras; default True                  |
| `created_at`  | DateTime tz        |                                             |

IVA no se almacena: se calcula `monto × tarifa / 100`.

### C. `motor/d150.py`

```python
def build_d150(db, cliente_id, periodo) -> dict:   # estructura PRECISA (Decimal)
def d150_ovi(preciso: dict) -> dict:               # vista entera estilo OVI-Tribu
```

`build_d150`:
1. **Ventas:** `build_resumen(db, cli, periodo, "venta")` → colapsa por tasa
   (`{tipo} {pct}` → `{pct}`, suma base/iva), exentas (`* Exento`), no_sujetas
   (`No Sujeto`). Suma las entradas manuales `rol="venta"`.
2. **Compras:** `build_resumen(db, cli, periodo, "compra",
   excluir_tipos={"TiqueteElectronico"})` → por tasa (crédito), exentas, no_sujetas,
   no_deducibles (`No Deducibles`). Suma las entradas manuales `rol="compra"`
   (deducible → crédito; !deducible → no_deducibles; tarifa 0 → exentas/no_sujetas).
   Informativo de tiquetes: consulta dedicada (n de comprobantes + suma de IVA).
3. **Totales** (precisos) y **liquidación**: `debito = Σ iva ventas`,
   `credito = Σ iva compras`, `neto = debito − credito`, estado.

`d150_ovi(preciso)`: por tasa `base_ovi = round(base)`, `iva_ovi = round(base × pct/100)`;
totales = suma de redondeados; estado desde el neto OVI.

**Estructura (precisa)** — la OVI es igual con enteros:
```
{
  "ventas":  {"por_tasa": {"1%": {"base","iva"}, ...}, "exentas", "no_sujetas",
              "total_gravadas", "total_impuesto", "total_general"},
  "compras": {"por_tasa": {...}, "exentas", "no_sujetas", "no_deducibles",
              "total_gravadas", "total_credito", "total_general",
              "tiquetes_excluidos_n", "tiquetes_excluidos_iva"},
  "liquidacion": {"debito_fiscal", "credito_fiscal", "impuesto_neto", "estado"}
}
```

### D. Schemas + endpoints (`schemas/entrada_manual.py`, `routers/entradas_manuales.py`, `routers/d150.py`)

- `POST /api/entradas-manuales` (201) — crea una entrada (valida rol ∈ {compra,venta},
  tarifa ≥ 0, monto ≥ 0). `GET /api/entradas-manuales?cliente_id&periodo[&rol]` —
  lista. `DELETE /api/entradas-manuales/{id}` — corrección.
- `GET /api/d150?cliente_id&periodo` → `{"preciso": <D150 serializado, Decimals como
  str>, "ovi": <D150 con enteros>}`.
- Todos protegidos por JWT.

## Flujo de datos

```
ingesta → comprobantes/líneas (compra/venta)
entradas manuales (POST) → tabla entradas_manuales
                                   │
GET /api/d150 → build_d150: build_resumen(venta) + build_resumen(compra, excl tiquetes)
                            + entradas manuales (rol) → colapso por tasa → liquidación
              → d150_ovi(preciso) → {preciso, ovi}
```

## Estrategia de pruebas (TDD)

Golden tests deterministas:
1. **`excluir_tipos`** — con un `TiqueteElectronico` y una factura, `build_resumen`
   excluye las líneas del tiquete.
2. **`build_d150` ingestado** — ingesta `fe_almacen_leon.xml` (compra 13%) → crédito
   13% refleja base 1858.40 / iva 241.59 (preciso) y su OVI; tiquetes excluidos del
   crédito (comprobante sintético `tipo_doc="TiqueteElectronico"`).
3. **Redondeo OVI** — base×tasa redondeado a entero; total = suma de redondeados.
4. **No Deducibles / No Sujeto fuera del crédito** — vía reglas 1B-5.
5. **Entradas manuales** — venta gravada→débito; venta exenta/no_sujeta; compra
   deducible→crédito; compra no deducible→no_deducibles; compra de subasta (gravada
   deducible).
6. **Liquidación + estado** — a_pagar / saldo_favor / cero.
7. **Endpoints** — `/api/d150` (preciso+ovi), CRUD de entradas manuales, 401/422.

Fixtures reales existentes + comprobantes/entradas construidos en el test.

## Riesgos / supuestos

- **Redondeo OVI**: se asume la fórmula del sistema viejo (`iva = round(base × tasa)`,
  totales = suma de redondeados). A confirmar en la reconciliación real.
- **Sin prorrata**: el crédito puede sobrestimarse para clientes con ventas exentas
  altas (caso Solis). Es fiel al sistema viejo y se documenta; la prorrata es una fase
  posterior.
- **Estado en frontera**: `estado` puede diferir entre la vista precisa y la OVI si el
  neto es ~0; el oficial es el de la vista OVI.
